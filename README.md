# claude-workflow

My personal Claude Code workflow — a reusable closed-loop engineering harness, packaged
as a **plugin** and distributed through this repo's own **marketplace**. The plugin ships a
*bootstrapper* that stamps a self-contained workflow into any repo, so each project ends up
owning its own engine — no rebuilding it by hand each time.

## Architecture: the engine lives in each project

```
plugin: harness-kit  (install once, available globally)
├── /harness          ← command: deterministic entry — drives the repo's local engine (Phases 0–9)
├── db-guard hook     ← plugin-global PreToolUse — asks before destructive DB ops (DROP/TRUNCATE/…)
├── harness-init      ← bootstrapper: explore repo → stamp the workflow into it
├── dev-init          ← directory / Docker / devlog scaffold
├── coding-standards  ← engineering operating principles
├── git-commit        ← commit-message style
└── grill-me          ← requirement interrogation (vendored from mattpocock/skills, MIT)

each project  (generated by harness-init — self-contained, no global dependency)
├── .claude/skills/harness-engineering/   ← the closed-loop ENGINE, local to this repo
│   └── LESSONS.md                         ← failure map (Phase 0 reads, Phase 9 writes)
├── CLAUDE.md                              ← thin: hard rules + decision boundary + routing
├── .claude/skills/<prefix>-<surface>/     ← domain skills, loaded on demand
├── docs/specs/                            ← intent map: five-doc spec per task (+ _templates/)
└── .claude/hooks/                         ← install-guard + retrospect-guard (settings merged)
```

Design point: the **engine is per-project**, not global. The plugin ships the *generator*
(`harness-init`) and the *template* for the engine; running it drops a tailored
`harness-engineering` into the repo. Every project is then self-contained — clone it and the
whole loop is right there, the same way a mature project keeps its own orchestrator.

**The loop** (run by each project's local `harness-engineering`, reusing the shared skills):

> Recall (lessons) → Intake (guardrails; `/grill-me` until a vague ask becomes a writable
> requirement) → **Spec (five docs = the contract**: requirement / design-when-UI /
> implementation / testing / fallback — mandatory, written before any code) → Dispatch
> → Ground (code wins for structure; the approved requirement wins for intent) → Implement
> → Verify (run `testing.md` literally) → Gate (`/code-review` + `/security-review`)
> → Report & **per-action git approval** (dead rule — no spec pre-approves a commit)
> → Retrospect (write LESSONS.md; a lesson hit twice gets promoted to a red line or hook)

Self-correction is bounded: ≤3 cycles per slice, then the circuit breaker trips — roll back
per `fallback.md` (with approvals) and escalate to the user with the failure trail.

## Install (marketplace + plugin)

```text
/plugin marketplace add jacksen-ng/claude-workflow      # register the marketplace
/plugin install harness-kit@jack-workflow               # install the bootstrapper + utilities
```

After install, the skills are **user-global** (stored under `~/.claude/plugins/cache/`).

## Using it in a new project

1. (Optional, greenfield) `dev-init` — lay down the directory / Docker / devlog scaffold.
2. `harness-init` — explores the repo and stamps in: a local `harness-engineering` orchestrator,
   a thin `CLAUDE.md`, and one domain skill per surface — for your review.
3. From then on, run any non-trivial task with **`/harness <requirement>`** — the deterministic
   entry that drives that repo's own `harness-engineering` loop (the skill also auto-triggers as a
   backstop).

## Update loop (how edits take effect)

Plugin installs are **copied to a cache**, so editing this repo does not take effect
automatically:

```text
# 1. edit a skill / template
# 2. bump "version" in plugins/harness-kit/.claude-plugin/plugin.json
# 3. push to GitHub
/plugin marketplace update jack-workflow
/reload-plugins
```

Note: bumping the plugin updates the *bootstrapper*, not orchestrators already stamped into
existing projects. Those are self-contained copies — upgrade them per the next section.

## Upgrading an existing project (stamped with an older engine)

Projects bootstrapped before v0.6.0 carry a self-contained older engine. The upgrade is the
same command as the first run — **re-run `harness-init` in that repo** — because its
reconcile path is the upgrader:

1. Update the plugin first: `/plugin marketplace update jack-workflow` → `/reload-plugins`.
2. Run `harness-init` in the project. Step 3 reads the engine's
   `<!-- harness-engineering template vX -->` marker; a v0.5.x engine is classified
   **UPDATE (propose)** listing the mechanism deltas (for 0.6.0: the five-doc spec gate,
   the `/grill-me` clarity gate, Recall/Retrospect + LESSONS.md, the circuit breaker, the
   Decision boundary, the retrospect-guard hook).
3. Approve the plan. Edits to existing files (engine, CLAUDE.md) are applied as approved;
   genuinely new artifacts (LESSONS.md skeleton, `docs/specs/` templates, the new hook) are
   additive and created directly.
4. Project assets are never regenerated: an existing `LESSONS.md`'s entries and any populated
   `docs/specs/` content are kept as-is — only the machinery around them is upgraded.
5. Committing the upgrade is, as always, a per-action approval (`/git-commit` when happy).

## Layout

| Path | Purpose |
|---|---|
| `.claude-plugin/marketplace.json` | marketplace manifest (name: `jack-workflow`) |
| `plugins/harness-kit/.claude-plugin/plugin.json` | plugin manifest (name: `harness-kit`, semver) |
| `plugins/harness-kit/skills/*` | the 5 bundled skills (harness-init, dev-init, coding-standards, git-commit, grill-me) |
| `plugins/harness-kit/commands/harness.md` | the global `/harness` command — deterministic loop entry |
| `plugins/harness-kit/hooks/` | plugin-global delete-safety: `db-guard.py` + `hooks.json` (PreToolUse) |
| `plugins/harness-kit/skills/harness-init/SKILL.md` | embeds Template A (engine) / B (CLAUDE.md) / C (domain skill) / D (install-guard) / E (LESSONS.md) / F (retrospect-guard) / G (spec docs) — the single source the bootstrapper stamps from |
