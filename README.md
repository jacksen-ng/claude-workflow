# claude-workflow

My personal Claude Code workflow — a reusable closed-loop engineering harness, packaged
as a **plugin** and distributed through this repo's own **marketplace**. Spin up a new
project or move to a new machine, and one `/plugin install` brings the whole workflow —
no rebuilding it by hand each time.

## Architecture: two layers + one loop

```
plugin: harness-kit  (install once, available globally in every project)
├── harness        ← generic closed-loop orchestrator
├── harness-init   ← project bootstrapper: explore repo → generate CLAUDE.md + domain skills
├── dev-init       ← directory / Docker / devlog scaffold
├── vibe-coding    ← engineering operating principles
└── git-commit     ← commit-message style

project layer  (each repo's own .claude/skills/, generated by harness-init)
├── CLAUDE.md            ← thin: hard rules + routing table
└── <prefix>-<surface>/  ← domain skills, loaded on demand (progressive disclosure)
```

**The loop** (orchestrated by `harness`, reusing the other skills):

> Intake (guardrails) → Plan (sprint contract) → Dispatch (load domain skill on demand)
> → **Ground (verify against live code** — map vs territory; the code wins) → Implement
> → Verify (separate generation from evaluation) → Gate (`/code-review` +
> `/security-review`) → Report & per-action approval

Design point: the generic **engine** (`harness`, installed once, works everywhere) and
the per-project **map** (`CLAUDE.md` + domain skills, generated fresh per repo) are kept
fully separate.

## Install (marketplace + plugin)

```text
/plugin marketplace add jacksen-ng/claude-workflow      # register the marketplace
/plugin install harness-kit@jack-workflow               # install the whole workflow
```

After install, the 5 skills are **user-global** by default and available in every project
(stored under `~/.claude/plugins/cache/`). Skills are namespaced as `harness-kit:<skill>`;
`harness` auto-triggers via its description, so the namespace mainly affects manual `/`
invocation.

## Using it in a new project

1. (Optional, greenfield) `dev-init` — lay down the directory / Docker / devlog scaffold.
2. `harness-init` — explores the repo → generates a thin `CLAUDE.md` + one domain skill
   per surface, for your review.
3. From then on, any non-trivial task flows through `harness` and the full loop.

## Update loop (how edits take effect)

Plugin installs are **copied to a cache**, so editing this repo does not take effect
automatically:

```text
# 1. edit a skill / template
# 2. bump "version" in plugins/harness-kit/.claude-plugin/plugin.json
# 3. push to GitHub
/plugin marketplace update jack-workflow
/reload-plugins
```

For local iteration without pushing, add the marketplace by local path
(`/plugin marketplace add ./` from this repo) and re-run the update/reload steps.

## Layout

| Path | Purpose |
|---|---|
| `.claude-plugin/marketplace.json` | marketplace manifest (name: `jack-workflow`) |
| `plugins/harness-kit/.claude-plugin/plugin.json` | plugin manifest (name: `harness-kit`, semver) |
| `plugins/harness-kit/skills/*` | the 5 bundled skills |
| `templates/CLAUDE.md.template` | thin CLAUDE.md skeleton (human reference; `harness-init` embeds the same) |
| `templates/domain-skill.SKILL.md.template` | domain-skill template (snapshot header + ground-truth anchors) |
